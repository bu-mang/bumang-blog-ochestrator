# 블로그 글 초안 작성

`bumang.xyz` 블로그에 올릴 글의 **초안**을 쓴다. 산출물은 마크다운 본문과 메타데이터 제안을 담은 파일 하나다. 발행(API 호출, 에디터 입력)은 사람이 한다 — 이 스킬은 거기까지 가지 않는다.

## 절차

### 0. 먼저 읽는다 (건너뛰지 말 것)

글을 한 줄이라도 쓰기 전에 아래를 **반드시 먼저 읽는다**. 톤앤매너의 정답은 머릿속이 아니라 이 파일들에 있다.

1. [BLOG_GUIDE.md](../../../BLOG_GUIDE.md) — 문체·구성·포맷 규칙 (원본)
2. [blog-references/](../../../blog-references/) 폴더의 실제 글들 — 문장 호흡·어휘·단락 끊는 법의 "족보"

가이드의 추상적 설명과 레퍼런스의 실제 문장이 어긋나면 **레퍼런스 문장을 따른다**. 레퍼런스를 읽지 않고 쓴 글은 십중팔구 톤이 어긋난다.

### 1. 주제와 재료를 확보한다

무엇에 대한 글인지, 어떤 사실·코드·수치를 담을지 사용자에게서 받는다. 부족하면 묻는다. 특히 다음이 있으면 글이 강해진다.

- 문제 상황과 그때의 고민 (왜 이걸 하게 됐는가)
- 실제로 한 조치 (코드·구조·의사결정)
- 결과 — 가능하면 **구체적 수치** ("응답 12초 → 0.4초", "OOM 0건")

기술 회고라면 코드 스니펫·에러 로그·before/after를 적극적으로 받아 둔다.

### 2. 본문을 쓴다

[BLOG_GUIDE.md](../../../BLOG_GUIDE.md)의 규칙을 그대로 지킨다. 요약하면:

- **평어체('-다')로만** 쓴다. '-습니다/-입니다' 금지.
- **불릿 포인트로만 글을 작성하지 않는다. 즉 남발하지 않는다.** 주장·근거·설명은 이어지는 문장으로 푼다. 불릿은 진짜 병렬 항목(단계·비교·체크리스트)에만.
- **문제 → 조치 → 효과**의 흐름으로, 소제목으로 박제하기보다 이야기처럼 잇는다.
- 원인과 결과를 풍부하게 서술한다. "A 했다, B 됐다"로 끝내지 말고 그 사이의 고민·선택지·작동 원리를 채운다.
- 포맷: 큰 절 `#`, 하위 `##`. 강조 `**굵게**`는 절제. 수치 비교는 표. 코드 블록은 언어 명시. 이미지는 앞뒤에 맥락 문장을 둔다. 주제 전환에 `---`.

### 3. 메타데이터를 제안한다

본문 끝에 발행용 메타데이터를 frontmatter로 함께 적는다. 필드 정의는 [BLOG_GUIDE.md](../../../BLOG_GUIDE.md) 4절과 백엔드 `PostEntity`(`bumang-blog-backend/src/posts/`)를 따른다.

```yaml
---
title: <글 제목>
previewText: <목록에 보일 한두 문장. 본문 첫 줄 복붙이 아니라 압축한 미끼 문장>
type: dev   # dev(개발) | life(일상). 기본 dev
category: <카테고리 이름 하나. 기존 것 재사용 우선>
tags: []    # 태그 여럿. 기존 태그 재사용 우선
readPermission: null   # null=전체공개 | guest | member | host (이상만 열람)
thumbnailUrl: null     # S3 업로드 이미지 URL. 없으면 null
---
```

- `category`/`tags`는 **기존 것을 먼저 재사용**한다. 새로 만들 거면 그 이유를 한 줄로 밝힌다. (콘텐츠 계층: `Group > Category > Tag`)
- `blockAudienceMap`은 보통 비운다. 특정 문단만 권한별로 가릴 때만 `block.id → groupId[]`로 채우는데, block id는 BlockNote가 부여하므로 초안 단계에선 "이 문단을 member 전용으로" 정도로 말로만 표시해 두고 비워 둔다.

### 4. 파일로 저장한다

초안은 레포 루트의 `drafts/` 폴더에 `drafts/<영문-슬러그>.md`로 저장한다. (폴더가 없으면 만든다.) frontmatter 메타데이터 + 본문 순서로 한 파일에 담는다.

`blog-references/`에는 저장하지 않는다 — 그 폴더는 이미 발행된 글의 톤 레퍼런스("족보")이지 초안 보관소가 아니다.

## 하지 않는 것

- 백엔드 API 호출이나 DB 쓰기로 **실제 발행하지 않는다.** 사람이 에디터에 붙여 넣는다.
- 커밋·푸시하지 않는다 (사용자가 명시적으로 요청할 때만).
- 잘 쓴 새 글이 레퍼런스가 될 만하면, 발행 후 `blog-references/`에 추가하자고 **제안**만 한다.
