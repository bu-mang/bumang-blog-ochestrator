# 어드민 API라고 방심하고 대규모 데이터를 한꺼번에 불러오면 안 되는 이유..(OOM)

## 들어가며
"어드민이니까 괜찮겠지." 사용자가 소수인 어드민 API를 만들 때, 성능 최적화에 대해 느슨해지기 쉽다. 사용자가 몇 명 안 되니까, 좀 무거운 쿼리를 날려도 괜찮을 거라 생각하게 된다.. 나도 그랬고 결과적으로 OOM(Out of Memory)의 무서움을 알게되었다.

![](https://bumang-blog-s3-storage.s3.ap-northeast-2.amazonaws.com/prod/thumbnails/1771469394136_external-1771469394088-129dfe69-125a-4a83-b9ed-3523d2aec9b4.png)
(그저 예시를 위한 이미지)

## 배경: 기능별 사용량 통계 API
Blai에는 기능별로 사용자들이 얼마나 쓰는지 조회하는 어드민 API가 있었다. 일별, 월별, 연별 단위로 사용량을 보여주는 기능이다. 데이터는 DocumentDB(MongoDB 호환)에 저장되어 있었고, 요청이 들어올 때마다 실시간으로 aggregation해서 응답하는 구조였다.
일별, 월별까지는 아무 문제 없었다. 문제는 **연 단위 조회**였다.

## 사건: 연간 조회 한 방에 OOM
하루에 약 8~10만 건의 사용량 데이터가 쌓인다. 단순 계산을 해보자.
> **10만 건/일 x 30일 x 12개월 = 약 3,600만 건**
연간 데이터를 한 번에 조회하면 수천만 건의 도큐먼트를 메모리에 올려야 했다. 게다가 해당 컬렉션에는 **조회 조건에 맞는 인덱스도 없었다.** 풀 스캔에 가까운 쿼리가 수천만 건의 데이터를 대상으로 실행된 셈이다.

결과는 OOM. 서버가 뻗었다.
DocumentDB에는 단일 aggregation에서 **100MB 이상의 메모리를 사용하면 자동으로 OOM을 발생**시키는 보호 메커니즘이 있다. 이 제한에 정확히 걸린 것이다.

이 이슈를 경험하기 전까지, db의 메모리를 전혀 의식하지 않았었고, 얼마짜리 인스턴스를 쓰고 있는지도 잘 몰랐다. 

**"어차피 어드민인데... 데이터가 많으면 타임아웃 좀 걸리는 정도겠지. 서비스가 뻗지는 않을거야."**
라며, 대규모 데이터를 로드하면서 타임아웃 가능성 까지는 생각했는데, 메모리 부족으로 서버가 뻗을거란 예측은 전혀 하지 못했었다. 어드민이긴 해도 상용 DB에서 데이터로 조회를 하고 있었기 때문에, 실제 사용자들에게도 장애가 나게 되었다. 처음 일으켜본 전시스템 다운타임이 일어나는 장애였다...

## 왜 처음부터 실시간 처리를 했는가
사실 처음부터 통계용 값을 미리 계산(pre-compute)해두자는 이야기가 있었다. 하지만 **서비스 초기**에 **UTC-KST 시간대 변환 이슈** 때문에 기존에 미리 생성해둔 통계 데이터에 오류가 많았다. 시간대 처리가 잘못되면 하루의 경계가 달라지고, 그 위에 쌓인 통계는 전부 틀어진다.
그래서 위클리 회의 때 **"차라리 실시간으로 raw 데이터에서 직접 계산하면 정확하지 않겠냐. 어차피 하루에 만건도 안 나오는데..."**는 결론이 나왔고, 실시간 aggregation 방식을 채택했었다. 그때는 문제가 없었기에 이 판단이 틀리지 않았다고 생각했다.

그러나 하루 10만건 정도의 분석이 실행되는 지금은 이야기가 다르다. 안 그래도 join 비용이 비싼 documentDB 상으로 실시간 aggregation을 많이 실행하는건 성능상으로 좋지 않다. 

## 해결: Pre-computed 통계 + allowDiskUse
OOM 이후, 두 가지 방향으로 개선했다.

### 1. 통계 전용 컬렉션 분리
매일 00시 정각에 배치 작업을 실행해서, 해당 일자의 통계 데이터를 별도 컬렉션에 미리 생성해두도록 했다. 조회 시에는 원본 데이터가 아니라 이 통계 컬렉션에서 가져온다.
- 원본 데이터: 수천만 건의 raw 로그
- 통계 데이터: 일 단위로 압축된 소수의 도큐먼트
연간 조회를 해도 365개 정도의 도큐먼트만 읽으면 되는 구조가 됐다.

### 2. allowDiskUse 옵션 적용
DocumentDB의 aggregation 파이프라인은 기본적으로 **100MB 메모리 제한**이 걸려 있다. 이 제한을 넘으면 OOM이 발생하는데, `allowDiskUse: true` 옵션을 설정하면 메모리 제한을 초과할 경우 **디스크를 활용**하도록 허용할 수 있다.
```javascript
db.collection.aggregate([
  { $match: { ... } },
  { $group: { ... } }
], { allowDiskUse: true })

```
pre-computed 통계로 전환했더라도, 배치 작업이나 불가피하게 대량 데이터를 다루는 쿼리에서는 이 옵션을 반드시 걸어두기로 했다. 100MB 제한의 존재를 모른 채 대량 쿼리를 날리면, 언제든 같은 일이 반복될 수 있기 때문이다.

### 3. 인덱스 추가
조회 조건에 맞는 인덱스도 추가했다. 애초에 인덱스가 없어서 풀 스캔이 발생했던 것도 느린 쿼리와 과도한 메모리 사용의 주요 원인이었다.

## 교훈

### "어드민이니까 괜찮겠지"는 위험한 생각이다
어드민 API는 사용자가 적지만, **오히려 그 때문에 한 번의 요청이 다루는 데이터의 범위가 클 수 있다. **전체 사용자 목록, 전체 기간 통계, 전체 로그 조회 등. **사용자가 적다는 건 요청 빈도가 낮다는 뜻이지, 요청 하나의 부하가 작다는 뜻이 아니다.**

### 데이터 규모를 미리 계산하라
"하루에 10만 건이면 1년이면 얼마지?"를 미리 계산했더라면 사전에 방지할 수 있었다. API를 만들 때 예상 데이터 볼륨을 반드시 따져봐야 한다.

### 실시간 처리 vs Pre-compute는 트레이드오프다
실시간 처리가 항상 정확한 건 맞다. 하지만 데이터 볼륨이 일정 수준을 넘으면 실시간 처리는 더 이상 선택지가 아니다. UTC-KST 같은 시간대 이슈가 있더라도, 그건 pre-compute 로직에서 올바르게 처리하는 게 맞다. 시간대 버그가 무서워서 수천만 건을 매번 실시간으로 읽는 건, 더 큰 문제를 초래한다.

### DocumentDB의 메모리 제한을 인지하라
DocumentDB의 aggregation은 기본 100MB 메모리 제한이 있다. 이걸 모르고 대량 쿼리를 날리면 예고 없이 OOM이 발생한다. 대용량 데이터를 다룰 때는 `allowDiskUse: true`를 반드시 설정하고, 가능하다면 데이터를 미리 가공해두자.

## 마치며
어드민 API는 눈에 덜 띄고, 사용자가 적고, 우선순위에서 밀리기 쉽다. 하지만 그렇기 때문에 더 방심하기 쉽고, 방심한 대가는 OOM이라는 형태로 돌아온다. "어드민이니까"라는 말 뒤에 숨기지 말고, 데이터 규모를 따지고, 인덱스를 걸고, 필요하면 미리 계산해두자.
