# 결제내역은 어떻게 서비스의 SSOT가 되었나
처음에는 단순한 결제 내역이었다.
누가 결제했는지, 어떤 상품을 결제했는지, 얼마를 냈는지, 결제가 성공했는지 정도만 알면 됐다. **결제로그는 말 그대로 결제의 결과를 남기는 기록이었다.**
그런데 서비스를 운영하다 보면 결제는 단순히 돈이 오간 사건으로 끝나지 않는다. 결제는 권한을 만들고, 이용 기간을 연장하고, 쿠폰을 소진시키고, 추천인 보상을 발생시키고, 대시보드의 매출 숫자가 된다. 어느 순간 결제로그는 단순한 로그가 아니라 여러 도메인이 의존하는 기준점처럼 작동하기 시작했다.
이 글은 그 변천사를 정리해보는 글이다.

### **1. 시작: 결제로그는 그냥 결제 내역이었다**
초기의 결제로그는 단순했다. 필요한 질문도 명확했다.
```typescript
누가 결제했는가?
어떤 상품을 결제했는가?
얼마를 결제했는가?
결제가 성공했는가?
언제 생성됐고 언제 만료되는가?
```

그래서 결제로그에는 이런 필드들이 있었다. (예시에는 모든 필드를 보여주진 않으며, 이름도 다르게 정의되어있다.)
```typescript
user_id
oid
tid
type // "이용권" | "강의"  | "블라이몰"
payment_type
payment_method
product_type
price_origin
price_cash
price_point
product_id

create_dttm
expire_dttm
payment_status
```

이 시점의 ProductLog는 결제 처리의 결과물에 가까웠다.
Product가 실제 사용 권한을 나타낸다면, ProductLog는 그 권한이 생긴 원인을 기록하는 보조 데이터였다.
```typescript
Product = 현재 사용 가능한 상품(이용권)
ProductLog = 그 상품(이용권)을 만들었던 결제 기록
```

아직까지는 역할이 크게 헷갈리지 않았다.

### **2. 그런데 결제는 돈만의 문제가 아니었다**
문제는 결제가 성공하는 순간, 서비스 안에서 여러 일이 연쇄적으로 일어난다는 점이다.
결제 성공은 단순히 payment_status를 success로 바꾸는 일이 아니다.
```typescript
이용권이 생성된다
기존 이용권이 연장된다
강의 수강 권한이 생긴다
포인트가 차감된다
쿠폰이 사용 처리된다
추천인 보상이 지급될 수 있다
매출 대시보드에 반영된다
```

결제는 돈의 흐름이면서 동시에 권한 변경 이벤트다.
그래서 결제로그에는 결제 처리 과정 자체를 기록하기 위한 필드가 붙기 시작한다.

```typescript
payment_process_stages
payment_failure_reason
approval_result
net_cancel_info
```

예를 들면 이런 식이다.

```typescript
payment_process_stages: [
  {
    stage: 'approval',
    status: 'success',
    timestamp: ...
  },
  {
    stage: 'product_creation',
    status: 'success',
    timestamp: ...
  }
]
```

이때부터 결제로그는 단순한 영수증이 아니라, 결제가 서비스 권한으로 바뀌는 과정을 추적하는 프로세스 로그가 된다.
결제 한 건을 보면 단순히 “돈을 냈다”가 아니라 “그 돈으로 어떤 처리가 어디까지 완료됐는가”를 알 수 있어야 했다.

### **3. 재결제 분석이 들어오면서 로그는 통계의 원천이 됐다**
서비스가 커지면 자연스럽게 이런 질문을 하게 된다.
```typescript
이 결제는 첫 결제인가? 기존 고객의 재결제인가?
재결제면 만료 전에 연장한 것인가? 만료 직후 돌아온 것인가?
오래 떠났다가 재가입한 것인가?
```

이때부터 결제로그는 분석 데이터의 원천이 된다.
```typescript
repurchase_type
estimated_end_dttm // 이전 결제로그의 estimated_end_dttm을 참고해서 repurchase_type을 계산하기 위한 필드
```

repurchase_type은 결제를 다음과 같이 분류한다.
```typescript
'first'  // 첫 결제
'fast'   // 만료 전 재결제
'slow'   // 만료 후 7일 이내 재결제
'rejoin' // 7일 초과 후 재결제
null     // 해당 없음
```
이 값은 단순한 표시용 값이 아니다. 대시보드에서 재구매율, 반복 구매자, 복귀 유저를 분석하는 기준이 된다.

여기서 ProductLog의 성격이 한 번 더 바뀐다.
처음에는 결제 결과를 기록하는 로그였다. 그런데 이제는 사용자의 구매 행동을 분석하는 통계 원천이 된다. 이 차이는 꽤 크다.

```typescript
Product = 지금 이 사람이 무엇을 쓸 수 있는가
ProductLog = 그 상품(이용권)을 만들었던 결제 기록 + 이 사람이 어떤 구매 여정을 거쳐 여기까지 왔는가
```

현재 상태는 Product가 설명한다.
과거 행동과 의사결정은 ProductLog가 설명한다.

### **4. 쿠폰과 추천인이 붙으면서 결제로그는 정산 가능한 영수증이 됐다**
결제 금액은 단순하지 않다.
**정가가 있고, 할인액이 있고, 쿠폰이 있고, 최종 현금 결제액이 있다. 추천인 코드가 들어오면 보상 지급 여부도 따라온다.**
그래서 결제로그에는 이런 필드들이 붙는다.
```typescript
target_coupons
used_coupons
price_coupon
discount_amount
referrer_user_id
referral_code
is_referral_rewarded
referral_rewarded_at
```

이제 결제로그는 단순히 “얼마를 냈다”가 아니라 “왜 그 금액이 되었는가”를 설명해야 한다.
```typescript
정가는 얼마였는가?
쿠폰으로 얼마가 할인됐는가?
포인트는 얼마나 사용했는가?
실제 현금 결제액은 얼마인가?
추천인 보상 대상인가?
추천인 보상은 이미 지급됐는가?
```

이런 정보는 현재 쿠폰 정책이나 상품 가격을 다시 조회해서 계산하면 안 된다. 결제 당시의 사실이 중요하다.
결제 시점에는 A라는 쿠폰 정책이 있었고, 지금은 B로 바뀌었을 수 있다. 상품 가격도 바뀔 수 있다. 추천인 보상 정책도 바뀔 수 있다.
그래서 결제로그는 정산 가능한 영수증이 되어야 한다.
```typescript
현재 정책이 아니라 결제 당시의 계산 흔적을 남긴다.
```

이건 문서형 DB라서 허용되는 타협이라기보다, 결제 도메인에서는 관계형 DB에서도 흔히 하는 선택이다. 영수증은 과거 사실의 스냅샷이어야 하기 때문이다.

### **5. 구독 결제가 들어오면서 로그는 청구 이력까지 품었다**
단건 결제와 구독 결제는 성격이 다르다.
단건 결제는 결제가 이루어지고 단 한 번의 스냅샷으로 필요한 정보들이 고정된다.
```typescript
사용자가 결제한다
결제가 성공한다
이용권이 생성되거나 연장된다
결제로그가 success가 된다
```

구독 결제는 다르다.
```typescript
빌링키가 있다
시스템이 정기적으로 결제를 시도한다
성공하거나 실패한다
회차가 증가한다
구독 상태가 갱신된다
```

그래서 결제로그에는 구독 관련 필드들이 붙는다.
```typescript
billing_subscription_id // 이 결제로그에 관련된 결제 정보 스키마(billing_subscription)
billing_cycle_number // 구독회차. billing_subscription 스키마에도 있지만, current_billing_cycle_number로 가장 최근의 것만 기록된다.
conversion_from_product // 이전 결제 이용권의 아이디. 구독 업그레이드 시 이전 구독 이용권의 정보를 남긴다. 업그레이드 결제 취소 시 원복하기 위한 필드
```

말했듯 BillingSubscription과 ProductLog의 역할은 다르다.
```typescript
BillingSubscription = 구독 계약의 현재 상태
ProductLog = 실제 청구와 결제 발생 이력
```

구독이 살아 있는지, 다음 결제일이 언제인지는 BillingSubscription이 설명한다.
하지만 실제로 돈이 청구됐는지, 몇 회차였는지, 실패했는지, 얼마가 결제됐는지는 ProductLog가 설명한다.
이때부터 ProductLog는 사용자 직접 결제뿐 아니라 **시스템이 만든 자동 결제 이벤트까지 품게 된다.**
결제로그는 이제 결제 버튼의 결과물이 아니라, 청구 이력의 원장이 된다.

### **7. 관리자 액션까지 들어오면서 운영 로그가 됐다**
실제 운영에서는 모든 결제가 PG를 통해서만 발생하지 않는다.
```typescript
관리자가 직접 이용권을 부여할 수 있다.\
결제를 취소할 수 있다.\
취소에 따라 이용권 일수를 차감할 수 있다.\
추천인 쿠폰을 회수할 수 있다.\
강의 수강생 권한을 조정할 수 있다.
```

그래서 결제로그에는 관리자 액션도 들어온다.
```typescript
cancelled_by: {
  admin_id: string;
  name: string;
  cancelled_at: Date;
}
granted_by: { ... }
```

이 값 자체가 잘못된 것은 아니다.
결제 상세 화면에서 “이 결제를 누가 취소했는가”를 바로 보여줘야 한다면 ProductLog에 요약 필드로 있는 게 자연스럽다.

### **8. 그래서 ProductLog는 무엇인가**
여기까지 오면 ProductLog는 더 이상 단순한 결제로그가 아니다.
지금의 ProductLog는 이런 역할을 동시에 한다.
```typescript
결제 이력
권한 변경 트리거
매출 집계 원천
재결제 분석 원천
쿠폰 정산 근거
추천인 보상 근거
구독 청구 이력
관리자 수동 처리 기록
```

사실상 결제 중심 이벤트의 SSOT처럼 작동한다.
정확히 말하면 ProductLog 하나가 모든 도메인의 진짜 원천이라고 보기는 어렵다. 구독의 현재 상태는 BillingSubscription이 갖고, 이용권의 현재 상태는 Product가 갖고, 쿠폰의 현재 상태는 Coupon이 갖는다.
하지만 돈이 오간 사건과 그로 인해 발생한 구매/권한/정산의 흔적은 ProductLog에 모인다.
그래서 운영자 입장에서는 자연스럽게 ProductLog를 먼저 보게 된다.

```typescript
이 유저가 왜 권한을 갖고 있지?
이 매출은 어디서 왔지?
쿠폰이 적용됐나?
추천인 보상은 나갔나?
구독 몇 회차 결제였지?
관리자가 수동으로 준 건가?
```

### **9. 수많은 도메인의 SSOT가 된 결제 로그, 괜찮은가?**
"원래 비관계형 문서 DB에선 정규화 안 하잖아." 라는 생각으로 이렇게 고봉밥 문서를 만들어왔다. 
그런데 이제 이 productlog에 더 많은 정보들이 쌓이면 내가 파악할 수 있을까...?

지금 고연차 개발자들에게 ProductLog를 보여줬을 때 정보량은 완전히 이상한 수준은 아니라고 들었다. 운영 중인 서비스에서 결제로그가 이렇게 커지는 건 흔한 일이다.
하지만 그대로 계속 키우면 위험해지는 지점에는 이미 들어와 있다.

문제는 필드 개수 자체가 아니다.
문제는 역할이 섞이는 것이라고 한다. 운영관리측면에서만 필요한 필드들 한 두 개씩 계속 끼우다간 이게 결제로그인지 운영 로그인지 헷갈릴 지경이 될 수 있다는 경고를 받았다....

예를 들어 쿠폰 사용도 사실 결제로그에는 요약 및 id만 남기고, 상세 이력은 **쿠폰 도메인**에서 관리할 수 있다.
```typescript
ProductLog {
  used_coupons: string[]
  price_coupon: number
}
```

비관계형 DB라도 언제 한 번은 필드들의 분리를 생각해볼 타이밍이 생겨야 할 것 같고, 오늘 이에 대해서 정리해봤다.
