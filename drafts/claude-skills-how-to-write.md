# 클로드 스킬(Claude Skills)이 뭐고, 어떻게 작성하는가

## 들어가며
클로드 코드를 쓰다 보면 같은 지시를 매번 반복하게 되는 순간이 온다. 나한테는 "블로그 글 써줘"가 그랬다. 이 블로그는 평어체로 쓰고, 불릿으로 토막 내지 말고, 문장으로 풀어서 써야 하는데, 그냥 시키면 십중팔구 "~습니다" 체에 개요만 나열한 글이 나온다. 그래서 매번 "가이드 먼저 읽고, 레퍼런스 글들 톤 보고 나서 써"라고 손으로 일러줘야 했다. 한두 번이야 괜찮은데 매 세션 똑같은 말을 반복하니 이게 일이다.

이런 "매번 같은 절차를 알려줘야 하는 일"을 한 번 정의해두고 재사용하는 게 **클로드 스킬(Claude Skills)**이다. 이 글에서는 스킬이 정확히 뭔지, 프롬프트나 MCP 같은 다른 개념과 뭐가 다른지, 그리고 실제로 블로그 글쓰기 스킬을 하나 만들어보며 알게 된 작성법을 정리한다.

## 스킬이란 무엇인가
스킬을 한 줄로 줄이면 **"특정 작업을 수행하는 방법을 담은, 클로드가 필요할 때 알아서 꺼내 쓰는 지침 패키지"**다. 어떤 일을 할 때 따라야 할 절차, 도구 사용법, 도메인 지식을 미리 적어두고 폴더 하나로 묶어둔다. 그러면 그에 맞는 요청이 들어올 때 클로드가 그 폴더를 열어 읽고 그대로 따른다.

실체는 의외로 단순하다. `SKILL.md`라는 마크다운 파일 하나가 중심이고, 필요하면 보조 스크립트나 템플릿 파일을 같은 폴더에 함께 둔다. 거창한 코드가 아니라 **사람이 읽는 설명서를 클로드가 읽게 만드는 것**에 가깝다.

```
.claude/skills/
└── write-blog-post/
    └── SKILL.md      ← 이 파일 하나가 스킬의 본체
```

## 그래서 프롬프트랑 뭐가 다른가
처음엔 "그냥 프롬프트에 적으면 되는 거 아닌가?" 싶었다. 차이는 **언제, 얼마나 컨텍스트에 올라오느냐**에 있다.

스킬의 핵심 메커니즘은 점진적 공개(progressive disclosure)다. 클로드는 평소에 스킬의 **이름과 설명(description)만** 가볍게 알고 있다. 본문 전체는 들고 있지 않는다. 그러다 사용자의 요청이 그 설명과 맞아떨어지는 순간, 그제서야 `SKILL.md` 본문을 통째로 읽어 들여 실행한다. 덕분에 컨텍스트를 거의 쓰지 않으면서도 수십, 수백 개의 스킬을 갖춰둘 수 있다. 매번 긴 지침을 프롬프트에 붙여 넣을 필요가 없다는 뜻이다.

비슷해 보이는 다른 개념들과 비교하면 역할이 더 또렷해진다.

| 개념 | 역할 |
|---|---|
| **스킬(Skill)** | "이 작업은 *이렇게* 한다"는 절차·노하우 |
| **MCP 도구** | 외부 시스템과 연결하는 실제 기능(API 호출, DB 조회 등) |
| **서브에이전트** | 독립된 컨텍스트에서 작업을 통째로 위임받아 수행 |
| **CLAUDE.md / 메모리** | 세션 내내 항상 떠 있는 배경 지식 |

스킬은 자주 MCP 도구나 서브에이전트를 **조합해서 지휘하는 쪽**이다. "데이터는 이 MCP로 가져와서, 이런 순서로 가공하고, 이렇게 정리해라" 같은 흐름을 묶어두는 게 스킬이라고 보면 된다.

## 직접 만들어본 블로그 글쓰기 스킬
백 마디 설명보다 직접 만들어보는 게 빠르다. 앞서 말한 "블로그 글 써줘" 문제를 스킬로 풀어봤다.

완성한 `SKILL.md`의 앞부분은 이렇게 생겼다.

```markdown
---
name: write-blog-post
description: bumang.xyz 블로그 글 초안을 톤앤매너에 맞게 작성한다.
  사용자가 "블로그 글 써줘", "이거 블로그로 정리해줘", "회고 글 써줘"
  처럼 블로그/포스트 작성을 요청하면 이 스킬을 사용한다.
  결과물은 마크다운 본문 + 메타데이터 제안을 담은 초안 파일이다.
---

# 블로그 글 초안 작성

## 절차

### 0. 먼저 읽는다 (건너뛰지 말 것)
글을 한 줄이라도 쓰기 전에 BLOG_GUIDE.md와 blog-references/의
실제 글들을 반드시 먼저 읽는다. 톤의 정답은 머릿속이 아니라
이 파일들에 있다.

### 1. 주제와 재료를 확보한다
### 2. 본문을 쓴다 (평어체, 문장 위주, 문제→조치→효과)
### 3. 메타데이터를 제안한다
### 4. drafts/ 폴더에 파일로 저장한다
```

전체는 더 길지만 뼈대는 이게 전부다. 위쪽 `---` 사이의 frontmatter가 클로드가 평소에 들고 있는 부분이고, 아래 본문이 발동됐을 때 읽히는 부분이다.

여기서 중요한 부분은 글의 톤을 설명하기 위해 blog-references/의 글을 읽고 오라는 점이다.
'링크'를 거는 것이다. 컨텍스트 문서의 중복 기입을 피하고, 동일한 맥락의 문서가 있으면 그걸 다시 읽고 오라고 하는게 좋다. 

## 스킬 작성에서 제일 중요한 건 description이다
실제로 만들어보고 가장 크게 와닿은 건, **스킬의 품질이 본문보다 `description`에서 갈린다**는 점이다. 본문을 아무리 잘 써도 description이 부실하면 클로드가 그 스킬을 **애초에 꺼내지를 않는다.** 발동되지 않는 스킬은 없는 스킬이다.

그래서 description은 두 가지를 분명히 적어야 한다. 첫째, **이 스킬이 무엇을 하는지.** 둘째, 그리고 이게 더 중요한데, **언제 발동해야 하는지.** 나는 "블로그 글 써줘", "이거 블로그로 정리해줘", "회고 글 써줘"처럼 사용자가 실제로 던질 법한 말들을 description 안에 그대로 박아 넣었다. 추상적으로 "블로그 작성을 돕는다"라고만 적는 것보다, 트리거가 될 구체적인 표현을 나열해두는 쪽이 훨씬 잘 잡힌다.

반대로, 언제 **쓰지 말아야 하는지**를 적어주는 것도 효과적이었다. "코드 작업이 아니라 글 콘텐츠 작업이다" 같은 한 줄이 엉뚱한 상황에서 스킬이 튀어나오는 걸 막아준다.

## 정리하며 — 스킬을 만들 때 챙길 것들
짧게 만들어본 경험을 정리하면 이렇다.

- **description에 발동 조건을 구체적으로.** 사용자가 던질 실제 표현을 그대로 넣는다. 스킬은 발동되지 않으면 의미가 없다.
- **이미 있는 문서는 복붙하지 말고 링크한다.** 스킬은 지식의 사본이 아니라, 지식을 올바른 순서로 불러오는 절차다.
- **본문은 절차로 쪼갠다.** "0단계: 먼저 읽기 → 1단계 → 2단계"처럼 클로드가 따라갈 순서를 명시한다. 특히 "무엇을 먼저 읽어야 하는지"를 0단계로 못 박는 게 효과가 컸다.
- **하지 말아야 할 것도 적는다.** 발행은 사람이 한다, 커밋은 시킬 때만 한다처럼 경계를 그어주면 스킬이 선을 넘지 않는다.

매번 같은 설명을 반복하고 있다면, 그건 스킬로 만들라는 신호다. 한 번 절차를 글로 정리해두면, 그 다음부터는 클로드가 알아서 그 절차를 따라온다. 사실 이 글도 방금 만든 그 스킬을 처음으로 돌려서 쓴 결과물이다. 톤이 어색하지 않게 읽혔다면, 스킬이 제 역할을 한 셈이다.
